# Архитектура tg-proxy

## Общая схема

```
Мерчант-группа 1  <──>  Userbot (активный аккаунт)  <──>  Саппорт-группа 1
Мерчант-группа 2  <──>  Userbot (активный аккаунт)  <──>  Саппорт-группа 2
                                    │
                              PostgreSQL
                         (маппинг message ID)
```

Один активный TelegramClient обслуживает все пары групп одновременно. При пересменке оркестратор отключает один клиент и подключает другой — маппинги в БД остаются общими для обоих аккаунтов.

## Компоненты

### Orchestrator

Управляет жизненным циклом воркеров. Запускает только один воркер в каждый момент времени. При наличии двух аккаунтов каждые 30 секунд проверяет расписание и при необходимости выполняет пересменку:

```
stop(текущий) → start(следующий, catch_up=True)
```

`catch_up=True` вызывает `client.catch_up()` — Telethon запрашивает у Telegram все апдейты, накопившиеся за время gap. Дедупликация на уровне БД (`ON CONFLICT DO NOTHING`) исключает двойную пересылку.

### Worker

Обёртка над `TelegramClient`. Регистрирует все хендлеры событий при первом старте (`_handlers_registered` флаг исключает повторную регистрацию при повторном подключении). Перед регистрацией хендлеров разрешает entity для всех мониторируемых чатов — без этого фильтр `chats=` в Telethon может не срабатывать при подключении с нового хоста.

### Хендлеры событий

Каждый тип события обрабатывается изолированно:

| Хендлер | Событие Telethon | Что делает |
|---------|-----------------|------------|
| `messages.py` | `NewMessage` | Пересылает текст и одиночные медиа |
| `albums.py` | `Album` | Пересылает медиагруппы как единый альбом |
| `edits.py` | `MessageEdited` | Редактирует зеркальное сообщение; если `MessageNotModifiedError` — зеркалирует текущее состояние реакций |
| `deletions.py` | `MessageDeleted` | Удаляет зеркальное сообщение |
| `reactions.py` | `Raw(UpdateMessageReactions)` | Зеркалирует реакции |

Все хендлеры обёрнуты в `try/except` в воркере — исключение логируется, процесс не падает.

### Маппинг сообщений

```sql
CREATE TABLE message_mappings (
    src_chat_id  BIGINT NOT NULL,
    src_msg_id   INTEGER NOT NULL,
    dst_chat_id  BIGINT NOT NULL,
    dst_msg_id   INTEGER NOT NULL,
    PRIMARY KEY (src_chat_id, src_msg_id)
);
```

Маппинг двунаправленный: `get_mirror(chat_id, msg_id)` ищет запись как по `(src_chat_id, src_msg_id)`, так и по `(dst_chat_id, dst_msg_id)`. Это позволяет обрабатывать события с обеих сторон без дублирования данных.

## Ключевые решения и trade-offs

### Один процесс, два клиента — вместо двух процессов

ТЗ предлагало управлять воркерами через `subprocess`. Выбран альтернативный подход: один процесс с двумя `TelegramClient` и оркестратором внутри.

Плюсы: проще деплой (один контейнер), общая память (нет IPC), прямой доступ к asyncpg-пулу из обоих воркеров.  
Минусы: падение процесса роняет всё. Компенсируется `restart: always` в Docker.

### Userbot, а не Bot API

Bot API не поддерживает отправку от имени обычного пользователя — сообщения имели бы атрибуцию бота. MTProto через Telethon позволяет отправлять сообщения как обычный аккаунт без каких-либо маркеров автоматизации.

### Реакции: два пути доставки

В супергруппах Telegram отправляет разные апдейты в зависимости от того, кто поставил реакцию:

- `UpdateMessageReactions` — для реакций на сообщения, отправленные самим userbot'ом
- `UpdateEditChannelMessage` — для реакций на чужие сообщения (приходит как редактирование)

Поэтому реакции обрабатываются в двух местах: `reactions.py` (основной путь) и `edits.py` (ловит `MessageNotModifiedError` и зеркалирует `msg.reactions`).

Снятие реакции мерчантом не реализуемо: Telegram не присылает апдейт userbot'у, если реакция снята с чужого сообщения. Это ограничение MTProto API, обойти его невозможно без бота с правами администратора.

### Альбомы через `events.Album`

`events.NewMessage` получает каждое фото из альбома как отдельное сообщение. `events.Album` ждёт все части медиагруппы и передаёт их списком. Без этого альбом разваливается на отдельные сообщения на стороне получателя. В `NewMessage`-хендлере сообщения с `grouped_id != None` пропускаются.

### PostgreSQL вместо SQLite

Маппинг должен переживать рестарты и быть доступным обоим аккаунтам одновременно. SQLite не подходит для конкурентной записи из asyncio. PostgreSQL с asyncpg даёт нативный async-интерфейс без ORM-оверхеда.

### Alembic для миграций

Выбран как де-факто стандарт для Python-проектов. Позволяет накатывать и откатывать схему без ручного SQL. Миграции применяются автоматически в `entrypoint.sh` — это упрощает деплой и исключает забытый шаг при обновлении.

### Human delay

Случайная задержка 1–4 секунды перед каждой отправкой имитирует время набора текста. Без неё userbot виден как автоматизация по скорости реакции. Диапазон настраивается в `config.yaml`.

### Структурированные логи

`structlog` с `JSONRenderer` пишет логи в stdout. Это совместимо с Docker (`docker compose logs`), journald и любой системой сбора логов (Loki, ELK). Каждое ключевое событие содержит структурированные поля — удобно для поиска и алертинга.

## Что оставлено на потом

- `catch_up` после сетевого обрыва внутри сессии (сейчас срабатывает только при рестарте или пересменке)
- Метрики (счётчик пересланных сообщений, latency) — при необходимости добавляется через Prometheus или structlog-процессоры
- Graceful shutdown — сейчас SIGTERM прерывает работу немедленно; при необходимости можно добавить ожидание завершения текущих хендлеров
