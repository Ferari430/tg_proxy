# Заметки по проекту (для себя)

Пишу на Go, Python знаю слабо — этот файл объясняет что и зачем в проекте.

---

## Библиотеки

### Telethon (`telethon==1.36.0`)
Главная библиотека. Реализует MTProto — протокол Telegram.
Позволяет работать как **обычный пользователь** (не бот), то есть от имени реального аккаунта.
Умеет: слушать события (новые сообщения, реакции, правки), отправлять сообщения, файлы, реакции.

Аналог в Go-мире: что-то вроде `gotd/td`, только значительно проще в использовании.

### asyncpg (`asyncpg==0.31.0`)
Драйвер для PostgreSQL. Работает асинхронно (не блокирует поток пока ждёт ответ от БД).
Пишем чистый SQL через `$1, $2` плейсхолдеры (как `?` в Go's `database/sql`).

Аналог в Go: `pgx`.

### Pydantic (`pydantic==2.7+`)
Библиотека для валидации данных. Описываешь структуру через классы — она автоматически проверяет типы и выбрасывает ошибку если что-то не так.

Аналог в Go: что-то вроде структур с тегами `json:"..."` + ручная валидация, только здесь это встроено.

### PyYAML
Парсит YAML файлы в Python-словари. Один вызов `yaml.safe_load(file)` — и получаешь данные.

### python-dotenv
Читает файл `.env` и кладёт переменные в `os.environ`. Нужен только локально — в Docker переменные передаются напрямую.

### structlog
Логгер с поддержкой JSON-формата. Вместо строк пишет структурированные записи:
```json
{"level": "info", "event": "message.forwarded", "src_chat": 123, "dst_chat": 456}
```
Удобно парсить в любой системе логов (Grafana, ELK и т.д.).

Аналог в Go: `zap` или `zerolog`.

### Alembic
Инструмент для миграций БД. Держит историю изменений схемы в файлах-миграциях.
Команда `alembic upgrade head` применяет все непримененные миграции.

Аналог в Go: `golang-migrate`.

### pytz
Работа с часовыми поясами. Нужен чтобы корректно определять текущее время в `Europe/Moscow`.

## Структура проекта

```
src/
├── core/        — конфиг, логгер, исключения (ни от чего не зависят)
├── db/          — работа с PostgreSQL
├── handlers/    — обработчики событий Telegram
├── worker.py    — один Telegram-аккаунт (клиент + хендлеры)
├── orchestrator.py — управляет воркерами, следит за расписанием
└── main.py      — точка входа, собирает всё вместе
```

Зависимости текут в одну сторону:
```
main → orchestrator → worker → handlers → db, core
```

## Файлы

### `config.yaml`
Основной конфиг приложения. Здесь прописываются аккаунты, пары групп, задержки.
Секреты (`API_ID`, `API_HASH`, `DATABASE_URL`) **не хранятся** в этом файле — они задаются через переменные окружения. В конфиге написано `"${API_ID}"` — это плейсхолдер, который при загрузке заменяется реальным значением из окружения.

### `.env` / `.env.example`
Файл с секретами для локального запуска. `.env` в git не коммитится (в `.gitignore`).
`.env.example` — шаблон, его коммитить нормально.

## `src/core/config.py`

Описывает структуру конфига и загружает его из YAML-файла.

**Классы (каждый — это структура данных как struct в Go):**

- `TelegramConfig` — `api_id` и `api_hash` для подключения к Telegram API
- `ActiveHours` — диапазон часов когда аккаунт активен (`start: 8, end: 20`)
- `AccountConfig` — один аккаунт: путь к session-файлу + расписание. Если `active_hours = None` — аккаунт работает 24/7
- `GroupMapping` — одна пара групп: `merchant_chat` (ID группы мерчанта) и `support_chat` (ID группы саппорта)
- `HumanDelay` — диапазон задержки в секундах перед отправкой. Дефолт: 1–4 сек
- `AppConfig` — корневая конфигурация, содержит всё выше. Здесь живут два метода:
  - `find_mirror(chat_id)` — получаешь ID чата, возвращает ID зеркального чата. Работает в обе стороны: мерчант→саппорт и саппорт→мерчант. Возвращает `None` если чат не в конфиге
  - `is_monitored(chat_id)` — просто проверяет, отслеживается ли этот чат (bool)

**Функции:**

- `_expand_env(value)` — приватная. Рекурсивно проходит по словарю/списку и заменяет `${VAR}` на значение из `os.environ`. Если переменная не найдена — оставляет плейсхолдер как есть
- `load_config(path)` — публичная. Читает YAML-файл, прогоняет через `_expand_env`, валидирует через Pydantic и возвращает `AppConfig`. Выбросит ошибку если структура не та

## `src/core/logging.py`

Настройка логгера.

- `setup_logging(level)` — вызывается один раз при старте. Конфигурирует structlog: формат JSON, вывод в stdout, уровень логирования (INFO/DEBUG/WARNING). Нужно вызвать **до** первого использования `get_logger`
- `get_logger(name)` — возвращает логгер для конкретного модуля. Обычно вызывается в начале каждого файла: `log = get_logger(__name__)`. Потом используешь `log.info(...)`, `log.error(...)` и т.д.

Использование (везде одинаково):
```python
log = get_logger(__name__)
log.info("message.sent", chat_id=123, msg_id=456)
# Выведет: {"level": "info", "event": "message.sent", "chat_id": 123, "msg_id": 456, ...}
```

## `src/db/pool.py`

- `create_pool(dsn)` — создаёт пул соединений к PostgreSQL. `dsn` — строка вида `postgresql://user:pass@host:5432/dbname`. Пул держит минимум 2 и максимум 10 соединений. Возвращает `asyncpg.Pool` — этот объект потом передаётся в `MappingRepository`.

Аналог в Go: `pgxpool.New(ctx, dsn)`.

## `src/db/repository.py`

Весь SQL в одном месте. Принимает пул соединений, делает запросы.

**`MessageMapping` (dataclass)** — структура данных, строка из таблицы:
- `src_chat_id` — ID чата откуда пришло исходное сообщение
- `src_msg_id` — ID исходного сообщения
- `dst_chat_id` — ID чата куда переслали
- `dst_msg_id` — ID пересланного сообщения

**`MappingRepository`:**

- `save(src_chat_id, src_msg_id, dst_chat_id, dst_msg_id)` — сохраняет маппинг в БД. `ON CONFLICT DO NOTHING` — если запись с таким `(src_chat_id, src_msg_id)` уже есть, молча пропускает (защита от дублей при catch_up)

- `get_by_src(src_chat_id, src_msg_id)` — ищет маппинг по исходному сообщению. Например: "это сообщение пришло из merch_group с id=100, куда оно было переслано?" → вернёт запись с `dst_chat_id` и `dst_msg_id`

- `get_by_dst(dst_chat_id, dst_msg_id)` — ищет маппинг по пересланному сообщению. Например: "это сообщение в support_group с id=200, откуда оно пришло?" → вернёт запись с `src_chat_id` и `src_msg_id`

- `get_mirror(chat_id, msg_id)` — удобная обёртка. Пробует сначала `get_by_src`, потом `get_by_dst`. Используется когда не знаешь с какой стороны пришёл запрос. Нужен для дедупликации и резолва реплаев

## `src/handlers/messages.py`

Единственный реализованный хендлер — пересылка текстовых сообщений.

**`handle_new_message(event, client, repo, cfg)`**

Вызывается когда приходит новое сообщение в одну из отслеживаемых групп.

Шаги внутри:
1. Если это системное событие (вступил в группу, закрепил сообщение и т.д.) — пропустить
2. Если нет текста или есть медиа — пропустить (медиа добавим позже)
3. Найти зеркальный чат через `cfg.find_mirror()`. Если не нашли — пропустить
4. **Дедупликация**: проверить в БД, не пересылали ли мы уже это сообщение. Защищает от двух ситуаций:
   - `catch_up` после пересменки может прислать события повторно
   - Сообщение, отправленное аккаунтом №1 в support_group, аккаунт №2 после смены видит как "входящее" — без проверки он бы переслал его обратно в merch_group
5. Если сообщение — реплай, найти ID соответствующего сообщения в зеркальном чате (через БД). Если не нашли (сообщение до старта системы) — отправить без реплая, не падать
6. Подождать случайное время (human delay)
7. Отправить сообщение в зеркальный чат
8. Сохранить маппинг в БД

**`_human_delay(cfg)`** — приватная. Просто `await asyncio.sleep(random от min до max)`.
`asyncio.sleep` не блокирует — пока этот хендлер ждёт, другие события продолжают обрабатываться.

## `src/worker.py` — класс `UserbotWorker`

Обёртка вокруг одного `TelegramClient` (один Telegram-аккаунт).
Принимает конфиг, репозиторий и параметры аккаунта. Знает как подключиться, зарегистрировать хендлеры и отключиться.

**`__init__`** — создаёт `TelegramClient`. На этом этапе соединения ещё нет, просто объект.
`session_file` — путь к файлу где Telethon хранит состояние сессии (токен авторизации, pts для catch_up). Аналог refresh token, только в файле.

**`start(catch_up=False)`**
1. Регистрирует хендлеры (только один раз, флаг `_handlers_registered`)
2. Подключается к Telegram
3. Если `catch_up=True` — запрашивает у Telegram все пропущенные события с момента последнего disconnect. Это используется при пересменке аккаунтов

**`stop()`** — отключается от Telegram. Важно: `disconnect()`, не `log_out()`. Сессия сохраняется, аккаунт просто уходит в оффлайн.

**`_register_handlers()`** — приватная. Подписывает хендлеры на события Telethon:
```python
client.add_event_handler(функция, тип_события)
```
`events.NewMessage(chats=monitored, incoming=True)` — слушать только входящие сообщения только в указанных чатах. `incoming=True` отфильтровывает сообщения, отправленные нами самими.

Хендлер обёрнут в `try/except` — если что-то пошло не так, логируем и продолжаем. Процесс не падает.

**`_get_monitored_chats()`** — возвращает список всех ID чатов (мерчанты + саппорты). Передаётся в фильтр хендлера.

## `src/orchestrator.py` — класс `Orchestrator`

Управляет воркерами. Знает расписание, запускает нужный аккаунт, следит за пересменками.

**`__init__`** — сохраняет конфиг и репозиторий. `self._tz` — объект часового пояса для MSK.

**`run()`**
1. Создаёт `UserbotWorker` для каждого аккаунта из конфига
2. Определяет какой аккаунт сейчас должен быть активен
3. Запускает его (`start(catch_up=False)` — первый старт, не нужен catch_up)
4. Если аккаунтов два — запускает `_schedule_loop()` как фоновую задачу
5. `await asyncio.Event().wait()` — блокирует навсегда (пока не придёт Ctrl+C или сигнал). Это стандартный Python-способ "крутиться вечно"

**`_schedule_loop()`** — крутится каждые 30 секунд:
1. Определяет какой аккаунт должен быть активен сейчас
2. Если тот же — ничего не делает
3. Если другой — останавливает текущий, запускает новый с `catch_up=True`

**`_get_active_index()`** — смотрит на текущее время в MSK и возвращает индекс нужного аккаунта (0 или 1).
Логика: если `start < end` (например 8–20) — обычный диапазон. Если `start > end` (например 20–8) — диапазон переходит через полночь.

## `src/main.py`

Точка входа. Запускается командой `python -m src.main`.

1. `load_dotenv()` — читает `.env` файл (локально). В Docker этот файл не нужен, переменные уже в окружении
2. `setup_logging(...)` — настраивает логгер
3. `load_config(...)` — читает `config.yaml`
4. `create_pool(dsn)` — подключается к PostgreSQL
5. `MappingRepository(pool)` — создаёт репозиторий
6. `Orchestrator(...).run()` — запускает всё

`asyncio.run(main())` — запускает асинхронную функцию. В Go этого не нужно, там горутины сами по себе. В Python нужно явно создать event loop.

## `migrations/versions/0001_initial.py`

Первая (и единственная пока) миграция — создаёт таблицу `message_mappings`.

**Схема таблицы:**
```sql
id          BIGSERIAL PRIMARY KEY   -- автоинкремент
src_chat_id BIGINT                  -- ID исходного чата
src_msg_id  INT                     -- ID исходного сообщения
dst_chat_id BIGINT                  -- ID зеркального чата
dst_msg_id  INT                     -- ID зеркального сообщения
created_at  TIMESTAMPTZ             -- когда создана запись
```

**Индексы:**
- `UNIQUE INDEX uq_src` на `(src_chat_id, src_msg_id)` — быстрый поиск по исходному сообщению + защита от дублей
- `INDEX idx_dst` на `(dst_chat_id, dst_msg_id)` — быстрый обратный поиск (по пересланному сообщению)

Применить: `alembic upgrade head`
Откатить: `alembic downgrade -1`

## `Dockerfile` и `docker-compose.yml`

`Dockerfile` — однослойный образ на `python:3.11-slim`. Копирует код, устанавливает зависимости, запускает `entrypoint.sh`.

`entrypoint.sh`:
```sh
alembic upgrade head  # применить миграции
python -m src.main    # запустить приложение
```
Миграции запускаются каждый раз при старте контейнера, но `upgrade head` идемпотентен — если всё применено, ничего не делает.

`docker-compose.yml` — два сервиса: `app` (наш код) и `postgres`. PostgreSQL стартует первым, `app` ждёт пока он будет готов (healthcheck через `pg_isready`).

`sessions/` монтируется как Docker volume — session-файлы Telethon сохраняются между перезапусками контейнера.

## Как запустить локально (без Docker)

```bash
# 1. Активировать venv
source .venv/bin/activate

# 2. Скопировать и заполнить .env
cp .env.example .env
# заполнить API_ID, API_HASH, DATABASE_URL

# 3. Применить миграции (PostgreSQL должен быть запущен)
alembic upgrade head

# 4. Запустить
python -m src.main
```

Первый запуск попросит ввести номер телефона и код из Telegram — так создаётся session-файл. После этого авторизация больше не нужна.
